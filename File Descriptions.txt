ROOT‑LEVEL FILES
─────────────────────────────────────────────────────────────────────────────────────
• .gitignore
  • Lists the files & directories that Git should ignore (e.g. *.pyc, venv/, instance/,
    secret_key.txt).  It has no runtime impact but keeps the repo clean.

• File Descriptions.txt   ← this file
  • Serves as the official “what‑is‑what” reference for every file in the
    repository.  The new description below includes the additional
    Sub‑Task feature and explains all key changes.

• LICENSE
  • GPL‑3.0 license text.  Required for open‑source compliance; has no
    runtime effect.

• README.md
  • Project overview, feature list, quick‑start instructions, and usage
    examples.  It is the primary onboarding document for developers.

• docker-compose.yml
  • Defines the local Docker service that builds the Flask image and
    runs the application on port 5000.  Useful for CI and containerised
    development.

• dockerfile
  • Builds the production Docker image from the repository.  It installs
    the dependencies, copies the source, exposes port 5000 and starts the
    Flask app with `python run.py`.

• promtps.txt
  • Human‑readable reference that lists key commands, environment
    variables and snippets.  It is a developer aid only.

• requirements.txt
  • Pin the exact Python packages required for the app:
    Flask, Flask‑Login, Flask‑WTF, Flask‑SQLAlchemy, Werkzeug, etc.

• run.py
  • Entrypoint that imports the Flask factory `create_app()` from
    `app/__init__.py`, creates the app instance and runs it on
    0.0.0.0:5000.

• secret_key.txt
  • Stores a single line of random text that is read by
    `app/config.py` to set Flask’s `SECRET_KEY`.  This file is
    intentionally omitted from Git to keep the key secret.



APP PACKAGE (app/)
────────────────────────────────────────────────────────────────────────────────────
The *app* package contains the core Flask application: configuration,
database models, forms, helper functions, view routes, static assets
(CSS/JS) and Jinja2 templates.  The following sections describe
every file in the package.

--------------------------------------------------------------------
app/__init__.py
  • Application factory that:
    – Instantiates extensions (SQLAlchemy, LoginManager, CSRFProtect).
    – Loads configuration from `app/config.py` (`DevelopmentConfig` /
      `ProductionConfig`).
    – Registers the single Blueprint `main` from `app/routes/__init__.py`.
    – Creates the database tables on startup (`db.create_all()`).
  • Returns the Flask app instance.
  • No changes were made to this file for the Sub‑Task feature.
  • New context processor `inject_todo_lists` now supplies `todo_lists` for all templates.


--------------------------------------------------------------------
app/config.py
  • Holds two configuration classes:
    – `DevelopmentConfig` – debug on, SQLite dev database, etc.
    – `ProductionConfig` – minimal config for production.
  • Reads `SECRET_KEY` from the `secret_key.txt` file.
  • Unchanged for the new feature.

--------------------------------------------------------------------
app/forms.py
  • WTForms definitions for user registration and login.
  • No change needed for sub‑tasks – the feature is handled entirely
    in routes and templates.

--------------------------------------------------------------------
app/helpers.py
  • Utility functions that drive the daily/monthly task views.
  • Functions such as `init_user_task_list`, `get_monthly_view`,
    `get_daily_status`, and `update_daily_status` remain untouched.
  • Sub‑tasks do not interact with these helpers.

--------------------------------------------------------------------
app/models.py
  • Core SQLAlchemy ORM definitions for the application.
  • **New additions**:
    – `SubTask` model – represents a child task belonging to a
      `TodoItem`.  It stores `name`, `completed`, `timestamp`,
      `due_date`, and foreign keys to the parent `TodoItem` and the
      owning `User`.
    – Relationship `subtasks` added to `TodoItem` (one‑to‑many,
      cascade delete‑orphan).
  • Existing tables (`User`, `TaskList`, `DailyTask`, `TodoList`,
    `TodoItem`) remain unchanged.  The new model is created on the
    next `db.create_all()` call.

--------------------------------------------------------------------
app/routes/__init__.py
  • Creates the Blueprint `main` (`bp = Blueprint('main', __name__)`).
  • Imports sub‑modules (`auth`, `tasks`, `todo`) so that all routes
    are registered.
  • No changes for sub‑tasks.

--------------------------------------------------------------------
app/routes/auth.py
  • Handles `/`, `/login`, `/register`, and `/logout`.
  • Uses `User` model and `LoginManager` for authentication.
  • No modifications required for sub‑tasks.

--------------------------------------------------------------------
app/routes/tasks.py
  • Provides `/daily` and `/monthly` views that display the
    recurring daily‑task list and a calendar grid.
  • Uses helpers from `app/helpers.py` for data processing.
  • Sub‑task logic is unrelated to this module – it stays the same.

--------------------------------------------------------------------
app/routes/todo.py
  • **Core change** – contains the full logic for the To‑Do list
    feature, extended to support sub‑tasks.
  • Major actions handled:
    * `todolists` – list creation / display.
    * `confirm_delete_list` – confirm & delete a list.
    * `todolist` – show a specific list and process:
      – add normal items (`action='add'`).
      – add a sub‑task (`action='add_sub'`).
      – complete a normal item (`action='complete'`).
      – complete a sub‑task (`action='complete_sub'`).
      – remove a normal item (`action='remove'`).
      – remove a sub‑task (`action='remove_sub'`).
  • When a parent item is marked completed, all its `subtasks` are
    marked completed at the same timestamp.  Sub‑tasks can also be
    completed individually; in that case they are only visually
    flagged (green background) and do not move to the Completed
    section.
  • Uses `db.joinedload('subtasks')` to eager‑load the relationship
    and avoid N+1 queries.
  • No other modules are affected.

app/static/css/monthly.css

--------------------------------------------------------------------
app/static/css/style.css

--------------------------------------------------------------------
app/static/css/task_overview.css

--------------------------------------------------------------------

--------------------------------------------------------------------
app/static/css/todo_list_manager.css

--------------------------------------------------------------------
app/static/js/theme.js
  • Handles the light/dark theme toggle.
  • Reads/writes `data-theme` on the `<html>` element and stores the
    choice in `localStorage`.  Unchanged for the sub‑task feature.

--------------------------------------------------------------------
app/static/js/todo.js
  • **Enhanced script** that now works with multiple
    “due‑date” toggles – both the main item form and any sub‑task
    form.
  • Uses a `data-target` attribute to link each checkbox to its
    corresponding date‑picker container.
  • On change it shows/hides the container and clears the input when
    unchecked.
  • Keeps the original behaviour for the top‑level add‑item form.

--------------------------------------------------------------------
app/templates/base.html
  • Main layout template:
    – `<head>` includes the CSS/JS assets, theme toggle button.
    – `<header>` contains the navigation bar and theme toggle.
    – `<main>` block is overridden by child templates.
    – `<footer>` shows the current year (provided by a context processor).
  • No structural changes for sub‑tasks.

--------------------------------------------------------------------
app/templates/auth/login.html
  • Renders the login form (`LoginForm`).
  • Displays flashed messages and a link to register.

--------------------------------------------------------------------
app/templates/auth/register.html
  • Renders the registration form (`RegistrationForm`).
  • Displays flashed messages and a link to log in.

--------------------------------------------------------------------
app/templates/tasks/daily.html
  • Displays the daily task table for a chosen date.
  • Checkboxes represent completion status; form posts back to
    `/daily` for bulk updates.
  • Unchanged by the sub‑task feature.

--------------------------------------------------------------------
app/templates/tasks/monthly.html
  • Calendar view that shows each day’s completion status.
  • Uses helper functions to build the month grid.
  • Unchanged by the sub‑task feature.

--------------------------------------------------------------------
app/templates/todo/confirm_delete.html
  • Confirmation page shown before deleting a To‑Do list.
  • Provides “Yes” and “Cancel” actions.

--------------------------------------------------------------------
app/templates/todo/lists.html
  • Overview page showing all lists for the current user.
  • Allows creating a new list and deleting existing ones.

--------------------------------------------------------------------
app/templates/tasks/task_overview.html
• Renders the Task‑Overview page – a yearly heat‑map for a single task.  
  – Includes a drop‑down selector populated with the current user’s daily tasks; selecting a task reloads the page.  
  – Builds a 26‑row × 14‑column grid where each cell shows the date in “MM/DD” format.  
    • Green cells = task completed on that day.  
    • Red cells = task existed but was not completed.  
    • Blank cells = task did not exist on that day (before its start date or after its end date).  
  – Uses the new CSS classes defined for the grid cells (`grid-cell`, `green`, `red`, `blank`).  
• Unchanged by the sub‑task feature.

--------------------------------------------------------------------
app/templates/todo/todolist.html
  • Core UI for a single To‑Do list:
    – Header with the list name and back‑to‑lists link.
    – Form for adding a new item (name, optional due date).
    – Three sections: Pending, Scheduled, Completed.
    – Each parent item shows:
      * name,
      * “Complete” and “Remove” buttons,
      * nested `.subtasks` block (if any).
    – Sub‑task block:
      * name,
      * optional due date (if the parent has one),
      * “Complete” and “Remove” buttons,
      * green background when completed.
    – Form to add a sub‑task under a parent item:
      * hidden until the “Due date” checkbox is checked,
      * uses unique IDs and `data-target` for the JS to toggle
        visibility.
    – Completed items section shows the parent and its completed
      sub‑tasks with timestamps.
  • The template uses standard Bootstrap‑style classes defined in
    `style.css` and includes the `todo.js` script for date‑picker
    toggling.

APP INSTANCE FOLDER
────────────────────────────────────────────────────────────────────────────────────
The `instance/` directory is created automatically by Flask.  It contains:

• `app.db` – SQLite database file (auto‑generated).
• `secret_key.txt` – Not part of the repo, but read by `config.py`.

--------------------------------------------------------------------
DATABASE MIGRATION NOTE
────────────────────────────────────────────────────────────────────────────────────
Adding the `SubTask` model requires a fresh database or a migration:
  1. Delete `instance/app.db` (or run an Alembic migration if used).
  2. Restart the application – `app/__init__.py` calls `db.create_all()`,
     creating the `sub_tasks` table.
  3. Existing users can start adding sub‑tasks immediately.

SUMMARY OF NEW FEATURE
────────────────────────────────────────────────────────────────────────────────────
• Users can create nested sub‑tasks under any To‑Do item.
• Sub‑tasks inherit the parent’s due‑date logic; if the parent has
  a due date, a sub‑task’s due date defaults to that date but can be
  set to any earlier date.
• Completing a parent automatically completes all its sub‑tasks.
• Completing a sub‑task only gives a green background; it stays on the
  parent list until the parent is completed.
• All actions are protected by CSRF tokens, logged‑in checks, and
  user‑scoped queries, ensuring no cross‑user data leakage.
• UI uses indentation, color cues, and a single JavaScript helper
  that handles multiple due‑date pickers.
